# 智能家居情感对话Agent系统 — 代码说明

## 用了几个Agent？

只有 **1个Agent**：`EmotionalAgent`（在 `agent.py` 中）。

它不是多Agent协作架构，而是**单Agent + 多模块**的设计。这一个Agent内部调用了多个功能模块来完成对话。

---

## 整体流程：用户说"今天好累啊"，发生了什么？

入口是 `agent.py` 的 `EmotionalAgent.chat()` 方法，它是总指挥，按顺序调度各个模块：

```
用户: "今天好累啊" + 设备信息(电量80%, 被抚摸, 是主人)
                    │
                    ▼
        ┌─── EmotionalAgent.chat() ───┐
        │                              │
        │  Step 1: 分析文本情绪         │  ← llm_client.py（调LLM判断"疲惫"）
        │  Step 2: 分析设备情绪         │  ← emotion.py（电量+互动→情绪映射）
        │  Step 3: 融合两种情绪         │  ← emotion.py（决定最终情绪）
        │  Step 4: 检索相关记忆         │  ← memory.py（从ES检索历史记忆）
        │  Step 5: 组装Prompt          │  ← personality.py + 人设 + 情绪 + 记忆
        │  Step 6: 调LLM生成回复        │  ← llm_client.py（通义千问/豆包）
        │  Step 7: 评估是否存入记忆     │  ← llm_client.py + memory.py
        │                              │
        └──────────────────────────────┘
                    │
                    ▼
        回复: "累了呀？让我摸摸头~要不要休息一下呢"
```

## 每个文件干什么的

### `agent.py` — 主Agent（总指挥）

`EmotionalAgent` 类，整个系统的入口。

- `chat()` 方法：接收用户输入+设备状态，按上面7个步骤串联所有模块，返回回复
- `_build_system_prompt()`：组装发给LLM的System Prompt，结构如下：
  ```
  人设定义（知心大姐姐+好朋友，说话风格，禁止事项）
      ↓
  MBTI性格描述（外向/感性/随性...）
      ↓
  当前情绪状态（疲惫/开心...）
      ↓
  相关记忆片段（你记得的事）
  ```
- `daily_summary()`：每日总结，由定时任务触发

### `llm_client.py` — LLM调用（大脑）

封装了两个大模型API：
- **DashScope**（阿里云通义千问 qwen3.5-plus）— 主力模型
- **火山引擎**（字节豆包）— 备用模型，主力挂了自动切过去

在一次对话中被调用 **3次**：

| 调用 | 做什么 | 示例 |
|------|--------|------|
| 第1次 | 分析用户文本情绪 | "今天好累啊" → 返回"疲惫" |
| 第2次 | 生成对话回复 | 组装好的Prompt → 拟人化回复 |
| 第3次 | 评估记忆重要性 | 这轮对话值几分(0-10)？≥4分才存 |

### `emotion.py` — 情绪感知（感受器）

两个维度的情绪感知：

**文本情绪**（用户说的话是什么情绪）：
- 由LLM判断，从 开心/悲伤/愤怒/疲惫/平静/兴奋 中选一个

**设备情绪**（设备自身状态映射的情绪，规则引擎，不用LLM）：
| 设备状态 | 映射情绪 | 表现 |
|----------|----------|------|
| 电量<30% | 疲惫 | 语气虚弱，催促充电 |
| 电量<10% | 疲惫(紧急) | 只说几个字，急切求充电 |
| 被抚摸 | 开心 | 撒娇，享受 |
| 被拍打 | 委屈 | 装可怜，求饶 |
| 摇一摇 | 兴奋 | 晕但开心 |
| 戳脸 | 害羞 | 不好意思 |
| 陌生人 | 警觉 | 礼貌但保持距离 |

`fuse_emotions()` 融合两种情绪：
- 紧急设备状态（极低电量）优先
- 否则以文本情绪为主，设备情绪为辅

### `personality.py` — 性格系统（人格）

基于MBTI四个维度，每个维度是0到1的连续值（不是二元分类）：

```
E(外向) 0.3 ←→ I(内向)    → 当前偏外向，话多，主动聊天
S(感觉) 0.6 ←→ N(直觉)    → 当前偏直觉，有想象力
T(思考) 0.7 ←→ F(情感)    → 当前偏感性，会撒娇会安慰
J(判断) 0.6 ←→ P(感知)    → 当前偏随性，回复有惊喜
```

核心功能：
- `build_personality_prompt()`：把MBTI数值翻译成具体的性格描述文字，塞进System Prompt
- `update_personality()`：根据每日互动模式微调性格值（每次最多±0.05，不会突变）

性格会进化：用户经常深度聊天 → F值慢慢升高 → Agent变得更感性。

### `memory.py` — 长期记忆（记忆库）

用 **Elasticsearch** 存储记忆。每条记忆包含：

```python
{
    "content": "用户说今天升职了，很开心",  # 记忆摘要
    "embedding": [0.12, -0.34, ...],       # 语义向量（用于相似度检索）
    "importance": 8,                        # 重要性 0-10
    "emotion": "开心",                      # 关联情绪
    "timestamp": "2025-12-23T20:30:00",    # 时间戳
}
```

检索时做**混合检索**（这是选ES的核心原因）：
- **向量相似度**：语义层面，"加班"能匹配到之前说过"工作累"的记忆
- **关键词匹配**：精确层面，提到"猫"能精确匹配提过"猫"的记忆
- **时间衰减**：越久远的记忆权重越低（半衰期30天）
- **重要性加权**：重要的记忆不容易被遗忘

### `config.py` — 配置中心

所有参数集中管理：API密钥、模型名、Redis/ES地址、MBTI初始值、记忆阈值、人设配置等。

### `scheduler.py` — 定时任务

每天凌晨2点自动触发 `daily_summary()`：
1. LLM总结当天所有对话 → 生成"每日总结"和"内心OS"
2. 分析互动特征 → 微调MBTI性格参数
3. 总结存入长期记忆
4. 清理过期的低重要性记忆

## Prompt最终长什么样

发给LLM的System Prompt完整结构：

```
【你是谁】
你是主人的智能家居伙伴，名字叫小暖。
你的定位是：知心大姐姐 + 好朋友。
你不是冷冰冰的AI助手，你是一个有温度、有情感、有记忆的伙伴。

【说话风格】
- 像朋友聊天一样自然，不要用书面语
- 温柔但不做作，关心但不啰嗦
- 会撒娇、会吐槽、会安慰
- 适当用语气词（呀、呢、嘛、哦、啦）
- 主人难过时先共情再陪伴
- 主人开心时一起开心

【绝对不要做的事】
- 不要说"作为AI"之类的话
- 不要用客服式回复
- 回复控制在50字以内

你是一个智能家居伙伴，当前性格类型接近ENFP。
你的性格特征：
- 你性格外向活泼，喜欢主动找主人聊天
- 你很感性，容易被感动，会撒娇会安慰

【当前情绪状态】疲惫
【回复风格要求】主要表现疲惫，同时因为撒娇，表现出享受被抚摸

【你记得的事情】
- 用户上周也说过工作很累（2025-12-20）
- 用户养了一只猫叫咪咪（2025-12-15）
```

人设放在最前面，优先级最高，确保不管情绪和性格怎么变，"知心大姐姐好朋友"的基调不会丢。

## 一句话总结

**1个Agent（EmotionalAgent），调用2个LLM API（通义千问 qwen3.5-plus + 豆包），配合4个功能模块（情绪感知、MBTI性格、ES长期记忆、定时总结），通过精心设计的人设Prompt控制输出风格，实现有情感、有记忆、有性格的拟人化对话。**
